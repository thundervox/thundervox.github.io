# Page 2 Stage

今となっては使う人のいない開発元からも見捨てられた脚本執筆ソフトウェアです。　これまで日本語化や移植など何度も挫折したり自然消滅しているので今回もどうなるかは不明です。それに今はかなりフリーウェアでも良い脚本執筆ソフトもありますからね。あくまで実用性や再商用化などの野心はなく暇つぶしや技術的興味のためものですので気長に期待せずお待ちください。

あくまで予定は予定に過ぎません。ほかにすることもありますからね。

## Very hard/difficulty

有償スペルチェッカーとシソーラスエンジンを採用している関係でオリジナルの開発環境を手に入れて、それらの機能を外してビルドしても機能面では同じになりませんし、メンテナンスを放棄された誰も触りたがらないソースコードなので大変困難です。ここ五年ほど状況はだいぶ良くなりましたが。

* かつては MSDN 以外での Visual C++ 6.0 は入手困難(eMbedded Visual Tools 3.0も公開終了のためWindows CEへ移植するのも無理)。違法ではなく普通に一般公開されていたものを組み合わせることで裏技でコマンドライン版コンパイラを再現することはできても MFC 6.0 がない。中古品を買っても配布ライセンスがない。かといって現行版の VisualStudio Community Edition では恐らくビルドするためにかなりの修正を要すると考えられます。

* デモ版の判定コードは各所に大量に埋め込まれているので除去するのはかなり手間です。段階的にクリーンアップしていくのが無難かもしれません。

* デッドコードのカタマリと化しているWintertreeライブラリをなんとかしないと面倒です。追記(2026年1月5日):暫定完了

* 初期段階では、Wintertree製スペルチェッカーライブラリはコメントアウトなりダミークラスを用意すればいいのでそれほど。シソーラスエンジンは自作なり探すしか無いかな。

## 今後の予定
* まずはドキュメントとリソースファイルの翻訳
* 次にちゃんと現行版のコンパイラでビルドできるようにする。WxWigetのOpenWatcom対応周りが不透明なのでClangなり他のC++コンパイラにすると思います。IDEはWideStudioは開発終了しているからCode::Blocksでいいかな。
* 自動車のレストアにしろソフトウェアの移植にしろ真っ先にすることはお掃除と現状確認なので当面はこちらを。
* ユニットテストの実装
* 配布サイトのリニューアル
* バージョン番号の変更: 2.x 系統へ
* インプットメソッドまわりの不具合を修正してインライン入力ができるようにする。
* Windows 9x などコンパイラやライブラリで対応していないOSのサポートは原則終了です。米国なら未だいるでしょうけど日本国内ですと PC-9801/Windows9xでPage2Stage を使っている本業の脚本家は見かけません。これではサポートするだけ余計なコストがかかり無意味です。
* WinSCP や Sakura Editor など Visual C++ 6.0 を過去に使っていて、いまでもマイグレーション後に開発が続いているソフトウェアのソースコードを参考に現行環境でビルドできるようにする。WinSCPのマイグレーション前後に対応するバージョンを探すのが大変そうですが。
* リファクタリングによるソースコードの可読性向上
* コーディングスタイルの変更(ハンガリアンからGoogle C++ Style Guideへ)
* Win32, ATL,MFC, Wintertree 依存からの脱却
* Windward時代の古い著作権情報、デモ版関連コード、スペルチェッカーまわりのWintertree依存コードの削除
* WinHelpからSphinxなどヘルプシステムの刷新
* FinalDraft など汎用ファイル形式への対応 (swsとrtf形式はコンバーター対応または段階的廃止予定)
* 長期的に拡張や保守のしやすく直交性の高い内部構造へ変更
* 完成後のソフトウェア名の変更 (未だWindward関連の商標権が残っているかもしれないので)
* PWCT 2.0 / Ring 1.x への移植はかなり先まで未定です (TrelbyのようにPython/Qt実装のようなこともできるはずですが)

最終目標として、日本国内のネットコミュニティではあまり具体例がなくネットの藻屑となりかけている MFC 6.0 → Linux へのレガシーマイグレーションに関する詳細解説のメモや記述を書いても、脱Windowsの流れをある程度早めることは難しいので期待しません。WindowsとLinux/XのAPI対応表みたいなものがあればある程度は楽になりそうですけどね。

ようするに、ドキュメントやリソースの翻訳から始まり、古くなったコードの書き換え、クロスプラットフォーム対応によるWindows対応の段階的廃止、最後にユーザーさんが自分好みにカスタマイズして長期保守できるよう PWCT と Ring への対応ってところですな。地味な作業ばかりですけど仕方ないですな。

## 覇権、成功、失敗

そんなの興味ありませんし、そういうことを言っているだけの人に用はありませんので。

多くの人にとって、このプロジェクトは1000％失敗として目に映るでしょうね。ですが、私にとっては成功しようが失敗しようがどうでもいいことです。ちゃんとビルドできるようにして少し手直して次にバトン渡すこと。それだけのトコです。神輿に乗るアホになりたくはありません。
